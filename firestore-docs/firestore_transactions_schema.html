Logika diskon
Composite indexes
Alur write transaksi
Logika resolusi diskon di item
Step 1 Cek wholesale pricing
Jika qty yang dibeli memenuhi syarat tier grosir di sku.wholesalePricing atau product.wholesalePricing, gunakan harga tier tersebut sebagai unitPrice.
Step 2 Cek diskon quantity
Jika owner mengatur diskon berbasis quantity (misal: beli 10 diskon 10%), sistem cek apakah qty memenuhi syarat. Diskon bisa berupa "percent" (nilai 0–100) atau "fixed" (potongan Rupiah per unit).
Simpan snapshot Harga final di item
Semua kalkulasi dilakukan di frontend sebelum write. Yang disimpan ke Firestore adalah hasil akhir:
unitPrice → harga sebelum diskon
discountType + discountValue → detail diskon untuk ditampilkan di nota
discountedPrice → harga per unit setelah diskon
subtotal → discountedPrice × qty
Composite indexes yang dibutuhkan
Riwayat transaksi per outlet wajib
tenantId ASC, outletId ASC, createdAt DESC
Untuk: kasir lihat riwayat transaksi hari ini di outletnya.
Laporan owner semua outlet wajib
tenantId ASC, createdAt DESC
Untuk: owner lihat semua transaksi dari semua cabang, diurutkan terbaru.
Filter by dateKey opsional
tenantId ASC, outletId ASC, dateKey ASC
Untuk: agregasi dashboard harian tanpa range query timestamp.
Urutan operasi saat kasir klik "bayar"
1 Validasi di frontend
Hitung semua subtotal, diskon, grandTotal, changeAmount. Pastikan semua item masih isActive: true.
2 Firestore batch write (atomic)
Dalam satu batch: (a) create dokumen transaksi baru, (b) kurangi stok setiap SKU fisik di products/{id}.skus[i].stock.{outletId} menggunakan FieldValue.increment(-qty).
3 Update aggregation (async)
Setelah batch sukses, update dokumen dashboards secara async — tambah revenue harian, tambah qty produk terlaris. Ini tidak blocking — boleh gagal dan di-retry.
4 Tampilkan nota & cetak
Frontend render nota dari data lokal (tidak perlu re-read Firestore). Trigger print via Web Bluetooth/USB jika tersedia.
