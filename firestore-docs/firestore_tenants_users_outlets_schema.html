Alur onboarding owner baru
1 Registrasi via Google / Email
Firebase Auth buat akun. Cloud Function onUserCreated trigger otomatis.
2 Cloud Function buat dokumen
Auto-create: tenants/{tenantId} dengan status trial, trialEndsAt = now + 28 hari. Auto-create: users/{uid} dengan role owner.
3 Set Custom Claims
Cloud Function set: { tenantId, role: "owner" } ke Firebase Auth token owner. Security Rules langsung aktif.
4 Onboarding form
Owner isi nama bisnis, tipe usaha, buat outlet pertama. Minimal 1 outlet wajib dibuat sebelum bisa mulai transaksi.
5 Siap digunakan
Owner langsung bisa tambah produk dan mulai transaksi. Timer trial 28 hari berjalan sejak step 1.
Alur owner menambah kasir baru
1 Owner input data kasir
Owner isi nama, email, dan pilih outlet penugasan di halaman manajemen karyawan.
2 Cloud Function buat akun kasir
Menggunakan Firebase Admin SDK: buat akun Auth dengan email kasir, set password sementara, buat dokumen users/{uid} dengan role cashier dan tenantId yang sama dengan owner.
3 Set Custom Claims kasir
Cloud Function set: { tenantId, role: "cashier" }. Kasir sekarang punya akses terbatas sesuai matriks akses.
4 Tambahkan ke outlet
UID kasir ditambahkan ke outlets/{outletId}.assignedCashiers outlet yang dipilih owner. Kasir bisa langsung login dan mulai transaksi di outlet tersebut.
Pindah outlet
Owner hapus UID dari outlet lama, tambahkan ke outlet baru. Cukup update 2 dokumen outlet — tidak perlu ubah dokumen user sama sekali.
Firebase Auth Custom Claims — kenapa penting
Apa itu Custom Claims?
Data kecil yang disimpan langsung di JWT token Firebase Auth. Bisa dibaca Security Rules tanpa membaca Firestore sama sekali. Contoh: { tenantId: "tenant_A7bR3k", role: "owner" }.
Kenapa tidak cukup pakai Firestore saja?
Jika role disimpan hanya di Firestore, Security Rules harus baca dokumen users/{uid} dulu sebelum memvalidasi setiap request. Itu 1 read berbayar per operasi. Custom Claims = 0 read, 0 biaya, lebih cepat.
Security Rules contoh penggunaannya
request.auth.token.role == "owner" — cek role
request.auth.token.tenantId == resource.data.tenantId — cek isolasi tenant
Keduanya dibaca dari token, bukan dari database.
