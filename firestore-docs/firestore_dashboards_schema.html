Strategi agregasi — kenapa tidak query langsung dari transactions?
❌ Cara naif
Query semua transactions bulan ini → hitung di frontend. 1000 transaksi = 1000 reads setiap owner buka dashboard.
→
✓ Cara kita
Dokumen agregasi diupdate setiap transaksi baru. Dashboard = 1–3 reads tanpa peduli berapa banyak transaksi.
Dokumen agregasi = pre-computed summary. Biaya write naik sedikit, biaya read dashboard turun drastis → tetap aman di Free Tier.

Identitas dokumen
▶
typestring"daily"
Tipe dokumen agregasi — untuk query filter.
Field ini memungkinkan query seperti where('type','==','daily') jika suatu saat semua agregasi digabung dalam satu koleksi besar.
▶
tenantIdstring"tenant_A7bR3k"indexed
Wajib untuk isolasi data multi-tenant. Security Rules cek field ini sebelum mengizinkan read.
▶
outletIdstring"outlet_JKTPST"indexed
Outlet mana yang direkap. "all" = konsolidasi semua cabang.
Untuk konsolidasi semua cabang, ada dokumen terpisah dengan outletId: "all". Dokumen ini diupdate bersamaan dengan dokumen per-outlet saat ada transaksi baru.
✓ Dashboard konsolidasi = 1 read dokumen "all". Drill-down ke cabang = 1 read dokumen outlet tertentu.
▶
dateKeystring"2025-07-14"indexed
Format YYYY-MM-DD — kunci utama filter harian.
Dipakai untuk query rentang tanggal: where('dateKey','>=','2025-07-01').where('dateKey','<=','2025-07-31') untuk laporan bulanan. Lebih efisien dari range query timestamp.
▶
monthKeystring"2025-07"indexed
Format YYYY-MM — untuk filter laporan bulanan sekaligus.
Memudahkan query where('monthKey','==','2025-07') untuk ambil semua dokumen harian dalam satu bulan tanpa range query. Lebih simpel dan efisien.

Alur update agregasi
Pola read dashboard
Estimasi biaya read
Alur update agregasi saat transaksi baru
1 Kasir klik bayar
Frontend validasi dan siapkan data transaksi.
2 Batch write atomic ke Firestore
Dalam satu batch (semua atau gagal semua):
• Create transactions/{id}
• Decrement stok di products/{id}.skus[i].stock.{outletId}
• Update tenants/{id}.lastTransactionAt
3 Cloud Function trigger (async)
onDocumentCreated('transactions/{id}') trigger otomatis:
• Increment dashboards/daily_* — revenue, transactions, revenueByPayment, dailyBreakdown
• Increment dashboards/monthly_* — sama
• Update topProducts di daily & monthly
• Cek low stock → update dashboards/lowstock_*
Note Async = tidak blocking kasir
Cloud Function berjalan di background setelah batch write selesai. Kasir tidak perlu menunggu agregasi selesai. Jika Cloud Function gagal, ada retry otomatis — data transaksi tetap aman.
Pola read saat owner buka dashboard
Dashboard hari ini — konsolidasi semua cabang 1 read
Baca dashboards/daily_tenantId_all_{today}. Dapat: total revenue, total transaksi, breakdown per metode bayar, top 10 produk.
Drill-down ke cabang tertentu 1 read
Baca dashboards/daily_tenantId_{outletId}_{today}. Data cabang spesifik.
Laporan bulan ini 1 read
Baca dashboards/monthly_tenantId_all_{monthKey}. Dapat: revenue bulanan, grafik harian (dailyBreakdown), top produk bulan ini.
Filter rentang custom (misal: 7 hari terakhir) max 7 reads
Query where('tenantId','==',tid).where('outletId','==','all').where('dateKey','>=','...').where('dateKey','<=','...'). Hasilnya array dokumen harian — jumlah reads = jumlah hari yang dipilih.
Low stock alert 1 read
Baca dashboards/lowstock_{tenantId}. Semua SKU menipis di semua cabang dalam satu dokumen.
Estimasi biaya read per bulan (Free Tier: 50.000 reads/hari)
Asumsi: 1 tenant, 2 outlet, owner buka dashboard 20x/hari
Dashboard buka: 20 × 3 reads (daily all + 2 outlet) = 60 reads/hari
Low stock check: 20 × 1 read = 20 reads/hari
Laporan bulanan (buka 5x/hari): 5 × 1 read = 5 reads/hari
Kasir lihat riwayat transaksi (50 transaksi/hari): 50 reads/hari
Kasir cari produk autocomplete: ~100 reads/hari

Total estimasi: ~235 reads/hari dari limit 50.000. Sangat aman di Free Tier.
Write cost per transaksi
1 transaksi = create trx + update stok (N SKU) + update daily + update monthly + update lowstock
Rata-rata: 5–8 writes per transaksi
Free Tier limit: 20.000 writes/hari
Aman hingga: ~2.500 transaksi/hari per tenant.
