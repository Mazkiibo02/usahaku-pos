# Product Requirements Document (PRD)
**Project Name:** Usahaku POS (Point of Sale & Business Analytics)
**Target Audience:** MSMEs, Store Owners, and Cashiers (Staff).
**Current Phase:** Phase 10 (SaaS Monetization & Production Deployment - Stabilized)

## Project Synopsis
Usahaku POS is a Multi-Tenant, Progressive Web App (PWA) Cashier (Point of Sale) application specifically tailored for Indonesian MSMEs. It delivers real-time transaction management, physical/digital thermal receipt printing, and comprehensive business analytics dashboards for store owners. The platform operates under a tiered SaaS Subscription model with strict resource gatekeeping and access control.

## SaaS Tier Structure & Pricing
To facilitate a Product-Led Growth (PLG) strategy, all newly registered businesses (tenants) automatically receive a **30-Day Free Trial** with a built-in capacity constraint of **maxOutlets: 2**. Once the trial expires or the owner opts to scale, they can select from the following subscription tiers:
* **1-Outlet Plan:** IDR 25,000 / month (Restricts platform access to exactly 1 active outlet).
* **2-Outlets Plan:** IDR 50,000 / month (Restricts platform access up to 2 active outlets).
* **4-Outlets Plan:** IDR 100,000 / month (Restricts platform access up to 4 active outlets).

## Feature Implementation Status (COMPLETED)
* **Tenant Authentication & Onboarding:** Successfully orchestrated via Firebase Auth and Cloud Functions. The system automatically provisions a 30-day trial metadata flag alongside `maxOutlets: 2` and `outletsCount: 0`.
* **Multi-Tenant Role-Based Access Control (RBAC):** Complete isolation between Owners and Cashiers enforced via Firebase Custom Claims (`tenantId` and `role`). All operational records are unified under the root `users` collection to maintain overall data schema integrity.
* **Cashier Synchronization Engine:** The `createStaffAccount` Cloud Function writes records directly to the root `users` collection, fixing the previous tracking bug for the "Total Cashiers" metric on the owner's dashboard.
* **Layout Guards & Cashier Routing:** Cashier roles are automatically hard-routed straight to the POS Register/Shift interface, completely blocking access to the owner's analytics graphs to eliminate potential "Missing or insufficient permissions" errors.
* **Core POS Register System:** Implemented active shopping cart calculations, product catalog management, shift accounting workflows (cash drawer float auditing), and real-time transaction checkouts.
* **Premium Bento Grid Landing Page:** Transformed the main route (`/`) into a premium marketing page, explicitly communicating the 30-Day Trial scheme while providing an integrated WhatsApp Business CTA bridge.
* **Midtrans Payment Gateway Core (Phases 1, 2 & 3):** Integrated the `SubscriptionLock.tsx` layout as a global reactive multi-tier paywall. Developed the `generateSnapToken` Cloud Function (2nd-Gen) for secure server-side itemized pricing verification. Deployed the `midtransWebhook` Cloud Function (2nd-Gen) equipped with SHA512 HMAC signature verification and a Firestore Transaction Guard. Built a Fail-Safe Manual Sync Feature (`checkPaymentStatus` 2nd-Gen) utilizing a 15-second debounce lock on both client-side (`localStorage`) and server-side (`lastCheckedAt`) to refresh security ID tokens via `getIdToken(true)` for instant web unlocking.
* **Anti-Spam & Payment Idempotency Guard:** Implemented pending invoice accumulation protection on the `/api/midtrans/token` endpoint. The server scans for active `PENDING` invoices under the current `tenantId` that are less than 24 hours old. If found, the system **reuses the existing snapToken**, locking the active Virtual Account number and preventing database invoice bloating. Supported by automated unit tests (`scratch/test-route-logic.ts`).
* **Midtrans Snap UI Callback Stabilization:** Refactored all `window.snap.pay` parameter interaction logic into arrow functions across the Settings panel and `SubscriptionLock.tsx`. This reactively resets the `isRenewing` state to `false` when the checkout iframe is dismissed (`onClose`, `onError`, `onPending`), preventing button UI freeze states.
* **Universal Multi-Path Thermal Print Architecture:** Formulated an ESC/POS binary data pipeline supporting multiple hardware channels: Web Bluetooth API utilizing a single-target query prioritizing the ISSC chip UUID (`49535343-fe7d-4ae5-8fa9-9fafd205e455`); Web Serial API for driverless Virtual COM USB cable printing on desktops; WebUSB API targeting the raw generic printer profile (`interfaceClass: 7`) split into standalone UI gestures to comply with Chromium sandbox rules; and Android RawBT Redirect via a custom intent scheme (`rawbt:base64,[DATA]`) to bypass mixed-content HTTPS restrictions on mobile Chrome.
* **Digital Receipt Image Rasterization Pipeline:** Integrated the client-side `html-to-image` conversion library to capture the DOM layout of the receipt and output a binary `image/png` Blob. Connected directly to the native **Web Share API (`navigator.share`)** to pop open the native OS Share Sheet for instant WhatsApp delivery, with an automatic download fallback for desktop devices.
* **Serverless Firebase Admin Credentials Resolution:** Fixed the Vercel serverless runtime deployment error (`Could not load the default credentials`) by configuring an explicit service account certification wrapper using environment variables and string-replacing multiline keys via `FIREBASE_PRIVATE_KEY.replace(/\\n/g, '\n')`.
* **Cashier Transaction Security & Access Rights Refinement (Firestore Rules Resolution):** Reconfigured the `firestore.rules` manifest and aligned frontend checkout payload queries so that accounts with a `cashier` status possess full operational rights to write transaction records (`create`) into the `invoices` collection as long as the document's `tenantId` matches their session Custom Claims. Restrained read queries so that cashiers can only query records tied to their explicit `outletId`, preventing lower-level staff from viewing global multi-branch corporate financial aggregates.
* **Product Catalog & POS Checkout Stabilization (Firestore CEL Syntax & Claims Alignment):** Resolved a critical production crash where the Firestore Query Optimizer statically blocked the product catalog list queries. Refactored `/products` read permissions to evaluate tenancy using a dedicated `isSameTenant()` helper, completely eliminating invalid non-CEL ternary operators (`? :`) from `isWritingToMyTenant()`. Additionally, integrated a forced token refresh mechanism (`getIdToken(true)`) inside `pos-service.ts` right before batch commits to ensure Owner/Cashier custom claims stay perfectly synchronized between client environments and Cloud Firestore.
